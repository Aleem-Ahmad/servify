SECTION 1 — PROJECT OVERVIEW QUESTIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q1. What is Servify and what problem does it solve?
A:  Servify is an online service marketplace platform that connects
    customers with verified home service providers (electricians,
    plumbers, carpenters, cleaners, painters, gardeners). The problem
    it solves is the difficulty people face in finding trusted,
    available, and fairly-priced local service workers. Traditional
    methods rely on word of mouth, which is unreliable and time-consuming.
    Servify digitizes this process with verified profiles, live booking,
    and feedback systems.

Q2. Who are the target users of Servify?
A:  There are three types of users:
    1. Customers — households or individuals who need a service
    2. Service Providers — skilled workers who want to get hired
    3. Admins — platform managers who verify providers and handle complaints

Q3. What makes Servify different from existing apps?
A:  Servify targets the local Pakistani market specifically. It supports
    Urdu language, uses Pakistani contact formats (CNIC, +92 phone),
    displays pricing in PKR, and is designed to work on mobile-first.
    It also includes a real-time OTP verification system before booking
    is confirmed and an admin verification step before providers go live.

Q4. What is the scope of this FYP?
A:  This is a frontend prototype with simulated backend using browser
    localStorage. It demonstrates the complete user journey for all three
    roles — customer, provider, and admin — with full UI, routing,
    bilingual support, dark/light mode, and responsive design.


SECTION 2 — TECHNICAL QUESTIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q5. What technology stack did you use?
A:  - Framework: Next.js (React-based)
    - Language: JavaScript (JSX)
    - Styling: Vanilla CSS (no Tailwind, no MUI)
    - Icons: Lucide React and React Icons
    - State Management: React Context API
    - Data Storage: Browser localStorage (mock backend)
    - Routing: Next.js App Router (file-based routing)

Q6. Why did you choose Next.js over plain React?
A:  Next.js provides file-based routing out of the box, which means
    each folder inside app/ automatically becomes a route. This made
    organizing the three dashboards (customer, provider, admin) clean
    and scalable. It also gives server-side rendering support for
    future backend integration, and has excellent performance optimizations.

Q7. Why did you not use a UI library like Material UI or Tailwind?
A:  We used plain CSS deliberately to show full control over the design.
    UI libraries add unnecessary bundle size for a prototype, and using
    custom CSS demonstrates deeper understanding of layout and styling
    principles. It also gave us complete freedom to implement the dark
    navy and orange brand theme without fighting library defaults.

Q8. How does authentication work in your app?
A:  Since there is no real backend, authentication is simulated using
    localStorage. On first load, dummy users are seeded into localStorage.
    When a user logs in, we search the users array for a matching email
    and password. If found, we save the user object as "currentUser" in
    localStorage. On every page load, we check this value to decide if
    access should be granted.

Q9. How did you implement role-based access control?
A:  Every protected route checks localStorage for a "currentUser" object.
    The user object contains a "role" field ("customer", "provider", or
    "admin"). If the role does not match the dashboard being accessed,
    the user is redirected. This is handled in the AuthContext and in
    layout files for each protected route group.

Q10. What is the booking flow for a customer?
A:  1. Customer opens the Booking/Complaint form (3-step wizard)
    2. Step 1: Select service category and describe the problem
    3. Step 2: Enter contact details (name, phone, address, live location)
    4. Step 3: Choose a provider from matching cards, generate OTP
    5. The OTP is auto-filled and displayed for verification
    6. Customer submits the booking — a confirmation message appears
    7. The booking is logged and the customer is redirected to the dashboard

Q11. What is the OTP system and why is it there?
A:  OTP (One-Time Password) is a 4-digit code generated when the customer
    selects a provider. It simulates a secure verification step — in the
    real app this would be sent via SMS. The provider would use this OTP
    to confirm they arrived and started the job. This prevents fraudulent
    bookings and ensures the right provider shows up.

Q12. How does the provider workflow work?
A:  1. Provider signs up with CNIC, category, and experience
    2. Admin reviews and approves the provider
    3. Once approved, provider appears in the marketplace
    4. Provider receives open service requests and can accept or reject them
    5. Provider completes the job and the customer can leave feedback
    6. Admin monitors all providers, can warn or block if needed

Q13. How did you implement the bilingual (English/Urdu) feature?
A:  We created a LanguageContext using React Context API. All displayable
    text strings are stored in two JSON files: en.json and ur.json.
    Each string has a unique key. Components use a t("key") function
    that reads the active language from context and returns the correct
    string from the matching JSON file. The choice is saved in localStorage.

Q14. How does dark mode work technically?
A:  A ThemeContext stores "dark" or "light" in state and localStorage.
    Components read this value and conditionally add a "dark" CSS class
    to their root element. CSS rules prefixed with ".dark" override the
    default light styles. This cascades down to all child elements.

Q15. How is the landing page structured?
A:  The landing page has 5 scroll-snap sections stacked vertically:
    Hero, Services, How It Works, Providers, and Footer/Contact.
    Each section is exactly 100dvh (full viewport height). On desktop,
    CSS scroll-snap ensures each scroll jump reveals exactly one section.
    On mobile, scroll snap is disabled so content flows naturally.

Q16. What is CSS scroll-snap and why did you use it?
A:  CSS scroll-snap is a browser feature that locks scrolling to predefined
    snap points. We used it so each section of the landing page fills and
    aligns perfectly with the viewport on desktop screens, creating a
    premium, app-like feel where sections slide in cleanly one at a time.

Q17. How did you handle responsiveness?
A:  We used CSS media queries at breakpoints 1024px, 768px, and 480px.
    On desktop, sections use fixed heights and scroll snap. On tablet,
    snap is disabled and sections use min-height. On mobile, content
    stacks vertically, font sizes reduce, and touch-friendly spacing
    is applied. The navbar changes from a top bar to a bottom tab bar
    on mobile.

Q18. How does the admin dashboard work?
A:  The admin can:
    - View all pending provider registrations and approve or deny them
    - View all existing providers with their ratings and status
    - Appreciate well-performing providers or issue warnings
    - Block providers who violate standards
    - Monitor customer complaints and feedback

Q19. What is localStorage and why did you use it instead of a real database?
A:  localStorage is a browser API that stores key-value pairs permanently
    in the browser (until cleared). We used it to simulate a backend
    database for this prototype. In a real production app, this would be
    replaced with a proper backend (Node.js, Django, or Firebase) and a
    real database (MongoDB, PostgreSQL). The frontend code is already
    structured so the API calls can be swapped in easily.

Q20. How would you connect a real backend in the future?
A:  We would replace all localStorage.getItem/setItem calls with
    fetch() or axios API calls to a REST API. The data shape would
    remain the same — only the source changes. Next.js's API routes
    or a separate Node.js/Express backend could serve the data.
    Authentication would switch to JWT tokens or session cookies.


SECTION 3 — DESIGN AND UX QUESTIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q21. Why did you choose dark navy and orange as your brand colors?
A:  Dark navy (#0a192f) conveys professionalism, trust, and reliability —
    important for a service platform. Orange (#ff7a00) is energetic and
    attention-grabbing, used as the primary action color. Together they
    create high contrast and a modern, premium visual identity that
    stands out from generic blue-and-white apps.

Q22. What UX principles guided your design?
A:  1. Consistency — same colors, fonts, and spacing throughout
    2. Clarity — clear labels, step indicators, and feedback messages
    3. Accessibility — bilingual support, readable font sizes, contrast
    4. Mobile-first — core flows work on small screens
    5. Feedback — success and error messages for every user action

Q23. Why did you use a 3-step wizard for the booking form?
A:  Breaking the form into steps reduces cognitive load. Instead of
    seeing a long overwhelming form, the user sees only relevant fields
    at each step. The step indicator shows progress and builds confidence.
    This pattern is common in premium booking apps like Uber and Airbnb.

Q24. How did you handle the provider cards design?
A:  Provider cards use a blob/organic shape (border-radius: 20% 50% 30% 40%)
    to make them visually distinctive and memorable. On hover they lift
    with an orange glow shadow, making the page feel interactive and alive.
    Cards show name, rating, category, and price so customers can decide
    quickly.

Q25. What design pattern is the login page based on?
A:  The authentication page uses a split-panel sliding design pattern.
    The left panel shows the form (login or signup) and the right panel
    shows an animated overlay with a call-to-action. When the user switches
    between login and signup, the panels slide and the overlay shifts.
    This is a modern pattern popularized by Dribbble UI designs.


SECTION 4 — CHALLENGES AND DECISIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q26. What was the biggest challenge you faced?
A:  Managing state across three dashboards (customer, provider, admin)
    without a real backend was complex. We had to design the localStorage
    data structure carefully so that all three dashboards could read and
    update the same "virtual database" consistently without conflicts.

Q27. If you had more time, what would you add?
A:  1. Real backend with Node.js and MongoDB
    2. SMS OTP via Twilio or Telenor API
    3. Real-time notifications (WebSocket or Firebase)
    4. Google Maps integration for live location tracking
    5. Payment gateway (JazzCash / EasyPaisa integration)
    6. Push notifications for mobile via PWA

Q28. Why did you not use Redux for state management?
A:  The app's complexity did not require Redux. React Context API is
    sufficient when state is shared between a moderate number of components
    and there are no highly complex update patterns. Redux adds boilerplate
    and learning overhead that was not justified for this prototype scope.

Q29. What security concerns would exist in a real deployment?
A:  1. Passwords stored in plain text in localStorage — would need hashing (bcrypt)
    2. No HTTPS enforcement — must add SSL in production
    3. JWT token expiry and refresh would need to be implemented
    4. Input sanitization to prevent XSS attacks
    5. Rate limiting on login attempts to prevent brute force
    6. CNIC and personal data would need encryption at rest

Q30. How did you divide the work in your team?
A:  (Answer based on your actual team division. Common answer structure:)
    One person handled the customer dashboard and booking flow.
    Another handled the provider dashboard and admin panel.
    Both collaborated on shared components (navbar, footer, services).
    Design decisions were made jointly, and CSS consistency was maintained
    through shared color variables and naming conventions.
